# Load test results — 2026-05-01

Single ledger-svc replica, single Postgres 18, k6 ramping-arrival-rate
scenario (Pareto-skewed: 80% of writes target the first 1% of accounts
within a tenant). Captured during the async-audit cutover validation.

## Topology under test

- 1× ledger-svc (Go 1.26, this repo)
- 1× Postgres 18 (Docker, 8GB shared_buffers, max_connections=400)
- 1× Redpanda (Kafka API)
- 1× Redis 8 (idempotency)
- k6 driving open-loop arrivals against the gRPC API

Single-host, no horizontal sharding, no read replicas.

## Scenario

```
stages:
  - 1000 RPS over 2m
  - 4000 RPS over 5m
  - 8000 RPS over 10m
  - 12000 RPS over 8m   ← peak
  - sustain peak  5m
total: 30m
account skew: pareto(0.8, 0.01)  // 80% of writes → first 1% of accounts
tenants: 1000
accounts/tenant: 100
```

## Headline result

| Stage time | Offered RPS | Sustained iters/s | 40001 retry-cap | Audit lag | Drain failures |
|-----------:|------------:|------------------:|----------------:|----------:|---------------:|
| 0:40       |       ~400  |              ~400 |             0 % |     0 s   |              0 |
| 10:22      |     ~6,500  |             5,347 |          17.3 % |     0 s   |              0 |
| 13:13      |     ~8,000  |             6,488 |          11.9 % |     0 s   |              0 |

The async-audit cutover lifted the throughput ceiling from the prior
~5K wall to 6.5K+ TPS sustained. The next ceiling is request-pool
saturation, not audit chain contention.

## What changed in this run

Two commits landed before this measurement:

1. **`feat(ledger): async audit`** (`112c849`) — request tx writes one
   row to `audit_pending` (atomic with the ledger row, no chain
   compute); a single-writer `AuditWorker` drains `audit_pending` →
   `audit_log` per-tenant. The `SELECT entry_hash FROM audit_log`
   inside the SERIALIZABLE block is gone.

2. **`fix(ledger): release pool conn on tx end`** (`dba9d5d`) — pgx 5.9
   quirk: `*pgxpool.Conn.BeginTx` returns a raw `pgx.Tx` whose
   `Commit/Rollback` doesn't return the slot to the pool. Wrapped in
   `releasingTx`. Without this, every InTx call leaked one conn; under
   load both pools hit MaxConns within seconds and every subsequent
   acquire timed out — symptom looked like a backend bottleneck but was
   actually a connection leak.

Order matters: the pool-leak fix is what unblocked the system; the
async-audit cutover is what removed the chain ceiling so the unblocked
system could deliver more than 5K TPS.

## Async-audit health

432K → 560K rows forwarded from `audit_pending` to `audit_log` over
the first 13 minutes:

- `audit_pending` queue depth: typically 4–51 rows in transit.
- `audit_pending_lag_seconds`: stayed at **0 s** through 6.5K TPS.
- Single drain worker, batch size 1000, polling at 50 ms.
- Worker pool acquired count: 1–2 of 5 throughout. Plenty of headroom.
- Zero drain failures.

The chain integrity guarantee is preserved (each tenant's hash chain
is computed in id-order, single-writer). The trade-off — audit_log
no longer atomic with the ledger row — is bounded operationally by
the lag metric; in this run the bound was 0 s.

## Where the new ceiling actually is

### Request pool saturation
- `pgxpool_acquired_conns` pegged at 250/250 from ~5K TPS onward.
- Mean tx commit time at this load: ~38 ms (250 conns × 26 commits/s = 6.5K TPS).
- This is PG itself pacing — buffer flush, WAL, index updates,
  SERIALIZABLE snapshot validation.
- `pgxpool_acquire_wait_seconds` p99 ≈ 50 ms, p99.9 ≈ 100 ms.
  Tight but not pathological.

### Hot-account 40001 contention
- 11.9–17.3% of attempts hit `ErrRetryCapReached` (SERIALIZABLE
  retry budget exhausted on 40001).
- Cause: Pareto skew. 80% of transfers target the first 1% of
  accounts → concurrent same-account transfers write-skew-conflict.
- The per-(tenant,method) circuit breaker shed the worst of these
  pre-flight (ratio fell from 17% → 12% as the breaker found its
  steady state); still material at peak load.

### Admission shedding
- 392K admission rejections by the 13-minute mark (~1.6K rejects/s).
- `grpc_admission_inflight` sat at 498 of 500. The cap is doing
  its job — better to fail fast at the edge than queue past pool
  capacity.

## What this proves

- The async-audit cutover delivered: chain compute is off the request
  path, lag stays at 0 s under load, no drain failures.
- The latent pgx pool-conn leak is fixed.
- The remaining ceiling is **scaling**, not architectural — pool
  sizing + PG `max_connections` tuning + (eventually) hot-account
  contention.

The "issues 1-8" plan in `scaling-issues-draft.md` predicted this
shape: removing audit chain pressure exposes the next bottleneck. We
landed on the predicted path.

## What this does NOT prove

- 10K+ TPS sustained on a single Postgres node. We hit 6.5K and the
  pool ceiling. Going higher needs the next cohort of changes
  (see `scaling-roadmap-10k.md`).
- Behaviour at multi-replica scale. Single-replica only.
- Behaviour with realistic tenant + account distributions. Pareto
  skew is harsher than typical production.
