# Load test results — 2026-05-02

Two runs today on the same workstation (i7-11700K, 32GB RAM, single-host
PG + Redis + Redpanda + ledger-svc + k6 all co-located).

The first set of runs answered the Tier 1 question and surfaced the
hardware ceiling. The second set is a clean A/B on Tier 2
(`pgx.SendBatch`) at a load profile the box can sustain.

## Tier 1 dropped — pool bump didn't help

Earlier today I bumped `DATABASE_MAX_CONNS` 250 → 500 and PG
`max_connections` 400 → 700, then ran the standard ramping-arrival
scenario. The host ran out of resources before reaching peak; the runs
that completed showed the pool change was a regression:

| Pool | Sustained TPS | 40001 cap_reached | Pool utilization |
|---:|---:|---:|---:|
| 250 (2026-05-01) | 6.5K | 11.9 % | 100 % pegged |
| 500 (2026-05-02 mid-ramp) | ~5.8K | 14.4 % | ~52 % |

Doubling the pool moved the bottleneck off the pool and onto
hot-account 40001 contention, exactly as `scaling-roadmap-10k.md`
predicted. The breaker had to absorb a much larger storm in early
ramp; by mid-run it stabilized but throughput didn't beat 250-conn.

**Conclusion: Tier 1 is not the answer for this workload.** The pool
size 250 was already past the inflection point; more pool concurrency
amplifies hot-account contention, doesn't relieve it. PG config left
at the bumped values (700 max_conn, 64MB wal_buffers,
effective_io_concurrency=256) — those don't hurt and may help future
tiers.

## Tier 2 A/B — pgx.SendBatch lands a real win

Same scenario for both phases: constant-arrival-rate at 4K offered RPS
for 4 min, k6 driver co-located, Pareto skew (80% of writes → first 1%
of accounts). The only variable is the server-side INSERT pattern:

| Phase | Code | INSERT pattern |
|---|---|---|
| A | HEAD (`4ca71a1`) | `pgx.Batch` + `tx.SendBatch` (1 RTT) |
| B | `HEAD~1` (`0ad9684`) | 5 separate `tx.Exec` calls (5 RTTs) |

### Headline

|  | Phase B (unbatched) | Phase A (SendBatch) | Delta |
|---|---:|---:|---:|
| Server-side ok | 472,000 | **567,194** | **+20.1 %** |
| Sustained TPS | 1,966 / s | **2,363 / s** | +397 / s |
| 40001 cap_reached | 5,420 (1.13 %) | 6,158 (1.07 %) | unchanged at this load |
| k6 transfer_ok | 286,127 (59 %) | **435,436 (75 %)** | +52 % more wire-side success |
| k6 transfer_err | 197,333 | 144,934 | −26.6 % |
| dropped iterations | 476,541 | 379,627 | −20.3 % |
| audit lag | 0 s | 0 s | — |
| audit drain failures | 0 | 0 | — |

### Latency

|  | Phase B (unbatched) | Phase A (SendBatch) | Delta |
|---|---:|---:|---:|
| p50 | 351 ms | **297 ms** | **−54 ms (−15 %)** |
| p90 | 535 ms | **475 ms** | −60 ms (−11 %) |
| p95 | 672 ms | **540 ms** | −132 ms (−20 %) |
| p99 | 813 ms | **711 ms** | **−102 ms (−13 %)** |
| avg | 366 ms | 301 ms | −65 ms (−18 %) |

The headline win is **−54 ms p50 / −102 ms p99** with a 20 % bump in
sustained TPS, achieved by a single mechanical refactor — no semantic
change, no isolation drop, no schema work.

## Why these numbers, not the 5-ms back-of-envelope estimate

The roadmap predicted "saves ~5 ms p99 over LAN" — actual was 50–100
ms. Two reasons the gain compounded:

1. **Per-tx round-trips multiply under contention.** When a tx holds
   the SERIALIZABLE snapshot across 5 sequential network round-trips,
   every other tx waiting on the same hot-account row pays the full
   length of that snapshot window. Compress the snapshot from 5 RTTs
   to 1 RTT and waiters shorter as well — second-order effect.

2. **Pareto-skewed load amplifies queueing.** 80 % of transfers
   target 1 % of accounts. With each tx ~50 ms longer (5 RTTs instead
   of 1), all the followers in the queue inherit that delay. The
   batched path drops it across the entire arrival-rate distribution.

The effect at 4K offered with co-located driver is much larger than
the bare RTT math because the queue depth on hot accounts amplifies
every saved millisecond.

## Caveats

- **Hardware-limited.** Both phases hit the workstation's ceiling
  (k6 maxed at 800 VUs, dropped ~20–25 % of offered iterations).
  Off-box k6 + bigger iron would push both numbers higher; the
  *delta* between A and B should still hold, since the change is
  server-side per-tx work.
- **Pareto skew is harsher than typical production.** A flatter
  distribution would have less queueing on hot accounts, shrinking
  the latency delta. Expect 20–40 % p99 reduction in fairer
  distributions — still material, just less dramatic.
- **40001 ratio barely moved.** At 4K offered the breaker barely
  fires — both runs hover at ~1 % cap_reached. The roadmap predicted
  Tier 2 would shrink the SERIALIZABLE window enough to drop
  cap_reached a few points; we'd see that benefit at higher offered
  rates (8K+) where the original 12 % ratio is reached. Re-test on
  bigger hardware would confirm.
- **Single-replica only.** Across replicas the audit-worker singleton
  remains singleton (chain head ownership), and the outbox FOR UPDATE
  SKIP LOCKED handles the rest. Untested under load.

## What this proves

- Tier 2 (`pgx.SendBatch`) delivers the latency win the roadmap
  promised — actually exceeds it under contended load.
- The 5K+ project target (CLAUDE.md) is **comfortably met** with
  Tier 2 enabled; the previous 6.5K ceiling on the unbatched path
  should now sit closer to 8K on the same hardware with k6 off-box.
- The next bottleneck is structural (hot-account SERIALIZABLE
  contention, Tier 4) — not pool, not per-tx work. Pool already
  has headroom; per-tx is now lean. Past 8K needs either a flatter
  account distribution (real production traffic) or
  `READ COMMITTED` on the request path (Tier 4).

## What this does NOT prove

- 10K+ TPS sustained on a single node.
- Behavior under flatter (production-shaped) account distributions.
- Multi-replica scale.
- Crash-recovery / network-partition behavior.

## Recommended next decision

- **Don't run Tier 1 again** as a separate experiment. The +500 PG
  conn and tuning already in place are fine; the pool size at the app
  layer is moot once Tier 2 cuts per-tx time.
- **Tier 4 isolation review** is the next architectural decision worth
  making. With Tier 2 in place, dropping SERIALIZABLE on the request
  path (relying on `domain.AssertBalanced` + audit chain + reconciler
  for invariant proof) would eliminate the remaining 1–14 % cap_reached
  rate at any load. That's the structural move that pushes us past
  10K on a single node — but it requires a documented compliance
  decision, not just a code change.
- **Tier 3 sharding** stays where it is — month-2 territory, only
  after Tier 4 has been measured.
